# rdf03: Endpoint Configuration Format

The JSON that drives AE RDF: **`config/app.json`** and the per-endpoint files it
references. This is the deploy-time contract — the authoring loop is *browse in
edit mode → tune per-type display → export → deploy*.

> Not to be confused with rdf00–rdf02, which describe the **parked offline**
> profiler/OWL pipeline. This spec is about the config the shipped browser loads.

## Where it lives

| File | Purpose |
|------|---------|
| `ae-rdf/public/config/app.json` | App-level config: name, logo, prefixes, and the list of endpoints. When it defines `endpoints`, the app runs in **config mode** (endpoint management locked). |
| `ae-rdf/public/config/endpoints/<slug>.json` | One endpoint's config. `<slug>` is the filename-safe endpoint name (`endpointSlug()` in `configExport.ts`). Same shape as one `endpoints[]` entry. |

Authoritative types: `ae-rdf/src/types/config.ts` (`AppConfig`, `ConfigEndpoint`)
and `ae-rdf/src/types/endpoint.ts` (`TypeConfig`, profiling types). The exporter
that writes these files is `ae-rdf/src/utils/configExport.ts` — it **omits empty
sections**, so a real file only contains what's set.

## `AppConfig` (app.json)

```jsonc
{
  "appName": "CORDIS Datalab",          // optional — header title
  "logoUrl": "https://…/logo.svg",       // optional — header logo
  "documentationUrl": "https://…",       // optional — replaces default docs link
  "prefixes": {                          // optional — GLOBAL prefix → namespace map
    "s66": "http://data.europa.eu/s66#", //   (prefixes are global, not per-endpoint)
    "skos": "http://www.w3.org/2004/02/skos/core#"
  },
  "endpoints": [                         // optional — presence ⇒ config mode
    "era-evr-kg-dev-noinf",              //   a STRING SLUG → config/endpoints/<slug>.json
    { "name": "CORDIS", "url": "https://…/sparql" } //   or an INLINE ConfigEndpoint
  ]
}
```

### `endpoints` — inline object or string slug

Each entry is **either** a full inline `ConfigEndpoint` **or** a string slug. A slug
is resolved by fetching `config/endpoints/<slug>.json` (a single `ConfigEndpoint`);
all slugs are fetched **in parallel** and spliced in before the rest of the app sees
the config. Adding an endpoint is then one file + one slug in the manifest — no
monolithic `app.json` to edit.

- A missing/invalid slug file is **dropped with a console warning** — it does not
  abort the load. The deployment still has *declared* endpoints, so it **stays in
  config mode** (a partial endpoint list, never a fallback to user-managed mode).

## `ConfigEndpoint`

```jsonc
{
  "name": "CORDIS Datalab",              // required
  "url": "https://cordis.europa.eu/datalab/sparql", // required
  "description": "…",                    // optional
  "auth": { "type": "basic" },           // optional — TYPE ONLY, never credentials (see below)
  "graph": { "quads": true, "defaultView": "own" }, // optional — query strategy axes
  "backend": "graphdb",                  // optional — profiler-only: property-listing query shape
  "infer": false,                        // optional — GraphDB inference: send infer=<value> on every query
  "deprecatedPredicates": ["http://www.w3.org/2002/07/owl#deprecated"], // optional — deprecation flags (profiler-detected)
  "suggestedLanguagePriorities": ["en"], // optional — label language order (default ["en"])
  "types": { /* type IRI → TypeConfig */ },          // display config (hand-authored)
  // ── profiling cache (generated by scripts/profile-endpoint.ts) ──
  "typeInventory":  [ { "uri": "…", "count": 78727 } ],
  "typeProperties": { "<type IRI>": { "ok": true, "sampled": false, "properties": [ … ] } },
  "subclasses":     { "<super IRI>": ["<sub IRI>", …] },
  "composition":    { "<class IRI>": [ { "uri": "<embed type>", "count": 42 } ] },
  "orphanCounts":   { "<embed type IRI>": 1495 },
  "profiledAt":     "2026-07-01T12:00:00.000Z"
}
```

### `graph` — query strategy (two orthogonal axes)

Drives `resolveGraphStrategy` (see rdf-overview "Graph model"). Either field unset
= *unknown* → the app queries a safe superset (named ∪ default).

| Field | Values | Meaning |
|-------|--------|---------|
| `quads` | `true` / `false` | Does the endpoint expose named graphs? |
| `defaultView` | `"own"` / `"merged"` | Is the default (no-`GRAPH`) view its own triples, or just a bag-y merge of the quads (never query it)? |

### `auth` — never bake credentials

Only `auth.type` (`"none" \| "basic" \| "apikey" \| "bearer"`) is exported.
Credentials are **never** written to a shared static file — the deployer supplies
them at runtime.

### `backend` — profiler-only query shape

`"graphdb" \| "virtuoso"` (unset ⇒ treated as `virtuoso`). Picks the shape of the
profiler's property-listing query: `graphdb` uses a flat join and skips the heavy
per-subject scans that time out on a GraphDB gateway; `virtuoso` uses the nested
quad-safe shape and profiles everything. **Read only by `scripts/profile-endpoint.ts`
— the running app ignores it.**

### `infer` — GraphDB inference control

Boolean. When set, **every** query (app *and* profiler) adds `infer=<value>` to the
POST body; omit the field to send nothing (endpoint default). The use case is a
second `…-noinf` config pointing at the same repository with `"infer": false`, to
browse asserted-only counts alongside the inferred view.

### `deprecatedPredicates` — deprecation flags

String[] of predicate IRIs. The profiler detects, from a seed of common flags (e.g.
`http://www.w3.org/2002/07/owl#deprecated`), which ones the endpoint actually asserts
as boolean `true`; the app **badges** any resource that asserts one of them true.
Unset ⇒ the app's built-in default set.

### Profiling cache

`typeInventory`, `typeProperties`, `subclasses`, `composition`, `orphanCounts`,
`profiledAt` are produced offline by `pnpm --filter ae-rdf profile-endpoint`.
They are **optional accelerators**: when present the app skips the corresponding
live query (instant Types sidebar, subclass tree, embed composition/orphan
badges); when absent it falls back to querying at runtime. `TypeProfile.ok`
(query succeeded) and `sampled` (derived from a `LIMIT` sample — list may be
incomplete) let a consumer decide whether to trust the property list.

### `typeProperties` — a `TypeProfile` per type

Keyed by type IRI. Beyond `ok` / `sampled`:

| Field | Type | Meaning |
|-------|------|---------|
| `ok` | boolean | The profiling query for this type succeeded. |
| `sampled` | boolean | The property **list** fell back to a `LIMIT` sample (may miss rare predicates); each listed property's counts are still measured over the full type. |
| `properties` | `TypeProperty[]` | The discovered predicates and their facts (below). |
| `embed` | `EmbedHints` | Embed candidacy — **profiler hint, not read by the app**. `selfMax`: the type's own biggest single-property fan-out (a wall of rows when inlined — a high `selfMax` is a flood trap, treat as NOT embeddable even at fan-in 1). `forward`: `[{via,max}]` for this type as an embedded **object** → `render:embed` + `embedVia:"<via>"`. `inverse`: same for this type as an embedded **subject** → `embedVia:"^<via>"`. Each list is `max`-ascending (best candidate first). |
| `blank` | boolean | **ALL** instances of the type are blank nodes ⇒ reachable only inline via a parent, so hide it + inline it. Set only when *exclusively* blank; a mixed type keeps its named instances navigable. |
| `bnodeCount` | number | Count of blank-node instances, recorded whenever non-zero (even for a mixed type, where `blank` is unset). |

### `properties[]` — per-property facts

Each entry is one predicate of the type, with the facts a full (non-sampled) scan
measured. Every field past `uri`/`count` is optional and omitted when it does not
apply.

```jsonc
{
  "uri": "http://data.europa.eu/949/role",   // the predicate IRI
  "count": 48521,                             // distinct (s,o) pairs — quad-safe basis
  "min": 1,                                   // per-instance cardinality: 0 = optional, ≥1 = required
  "max": 21,                                  // largest per-instance count (1 = single-valued)
  "nodeKinds": { "iri": 48521 },              // value kinds over the SAME pairs (Σ === count)
  "ranges": [                                 // object classes of IRI values (distinct s,o pairs)
    { "uri": "http://data.europa.eu/949/OrganisationRole", "count": 42117 },
    { "uri": "http://data.europa.eu/949/VehicleKeeperMarking", "count": 6404 }
  ],
  "untyped": 0                                // IRI-object pairs whose object has no rdf:type
}
```

A literal-valued property instead carries `maxLen`, `datatypes` and (for langString)
`languages`:

```jsonc
{
  "uri": "http://www.w3.org/2004/02/skos/core#definition",
  "count": 111,
  "min": 1, "max": 1,
  "maxLen": 145,                              // longest literal — display size hint
  "nodeKinds": { "literal": 111 },
  "datatypes": [
    { "uri": "http://www.w3.org/1999/02/22-rdf-syntax-ns#langString", "count": 111 }
  ],
  "languages": { "en": 111 }
}
```

| Field | Type | Meaning |
|-------|------|---------|
| `uri` | IRI | The predicate. |
| `count` | number | Distinct **(s,o) pairs** for this predicate over the type's instances — the quad-safe basis every other pair-count here is comparable with. |
| `min` | number | Smallest per-instance occurrence. `0` ⇒ some instance lacks the property (optional); `≥1` ⇒ required. |
| `max` | number | Largest per-instance occurrence. `1` ⇒ single-valued/functional; `>1` ⇒ multi-valued. |
| `maxLen` | number | Longest **literal** value (absent/0 for IRI-only properties). Size hint: high ⇒ long prose (suggest `capWidth`); many short repeats ⇒ suggest `columns`. |
| `nodeKinds` | `{iri?,literal?,bnode?}` | Value node-kind distribution over the same distinct (s,o) pairs — **zero kinds are omitted**, and the invariant `Σ === count` holds. Its **presence (even `{}`) is the "facts measured" marker** `--resume` reads. |
| `datatypes` | `UriCount[]` | Literal datatype distribution — full IRIs, count-descending. An `rdf:langString` entry ⇒ language-tagged text. Omitted when the property has no literal values. |
| `languages` | `{tag: count}` | Language-tag → count over langString values, keys sorted. Omitted if none. |
| `ranges` | `UriCount[]` | Object **classes** of IRI values, counted in distinct **(s,o) pairs** (so directly comparable with `count` / `nodeKinds.iri`), top 20 by count. A multi-typed object counts **once per class**. Omitted when there are no typed IRI objects. |
| `untyped` | number | Distinct (s,o) pairs whose IRI object has **no** `rdf:type`. **Always written (even `0`) once `ranges` was measured** — its presence is the "ranges measured" marker (both `ranges` and `untyped` are absent when the ranges step timed out). |

Invariant: `Σ ranges + untyped ≥ nodeKinds.iri`, with **equality when objects are
single-typed**. Worked micro-example — `nodeKinds.iri: 39`, `ranges: [ConceptScheme → 34]`,
`untyped: 5`: reads directly as *34 of the 39 IRI objects are ConceptSchemes, 5 are
untyped* — 34 + 5 = 39, no double-count, no guessing.

### Profiler workflow

```bash
node scripts/profile-endpoint.ts <endpoint.json> [--fast|--resume]
```

Writes the profiling cache (`typeInventory`, `typeProperties`, `subclasses`,
`composition`, `orphanCounts`, `profiledAt`) back into the config; hand-authored
`types` is preserved.

- **Credentials** come from `SPARQL_USER` / `SPARQL_PASS` env only — never the file.
- The config is **checkpointed atomically** (write `.tmp`, rename over) after **every
  measured property** and every facts step, so a killed run loses at most the step
  in flight.
- **`--resume` is per-property**: it reuses each type's prior property **list** and
  re-queries only the properties still missing a count *or* the facts/ranges marker
  fields above (an old counts-only config gets facts-only queries) — nothing already
  measured is re-run.
- **`--fast`** = property **lists only**, no per-property measurements (for huge
  endpoints).
- A plain full run **re-measures everything** but **reuses an existing
  `typeInventory`**; reset the file to a `{name, url}` stub to force re-discovery of
  the inventory too.

## `TypeConfig` — per-type display (the hand-authored part)

Keyed by type IRI under `types`. Every field is optional; an empty config is
pruned on export. This is what edit-mode authoring writes.

| Field | Type | Effect |
|-------|------|--------|
| `render` | `"link" \| "embed" \| "label"` | How an object of this type renders when referenced. `link` (default): clickable, shown by its label. `embed`: inline its properties as rows (small **value objects** only — see AE RDF CLAUDE.md). `label`: identity text only, no navigation. |
| `embedVia` | predicate IRI | Only inline (when `render:embed`) where the object is reached via **this** owning predicate. Stops a multiply-referenced entity from inlining everywhere. |
| `sidebar` | `"show" \| "hide" \| "pin"` | Types-sidebar treatment. `pin` floats it to the top; `hide` removes it from browsing. |
| `group` | string | Collapsible sidebar group label (e.g. `"Ontology"`). |
| `order` | predicate IRI[] | Field display order for a resource of this type. Listed predicates first (in this order); the rest fall to the default priority order. Authored by drag-reorder. |
| `hide` | predicate IRI[] | Predicates hidden from the detail view. Removed entirely in normal mode; greyed in edit mode. "Show hidden fields" (settings) reveals them read-only. |
| `label` | predicate IRI[] | Predicates whose values **compose** the display label (joined with ` · `, in list order). Overrides the default heuristic (`prefLabel → title → rdfs:label → …`). A URI-valued field resolves to the referent's own label, walked up to 3 hops (e.g. `GrantPayment → hasRecipient → OrganisationRole → isRoleOf → Organisation`). Unset ⇒ default heuristic. |
| `foldAfter` | predicate IRI | When inlined (`render:embed`), show rows up to and including this predicate; fold the rest behind "Show more". Unset ⇒ show all. |
| `groupByType` | predicate IRI[] | Predicates whose object lists render **grouped by the object's type** (a subheading + count per type) instead of a flat list — for long, mixed-type relations (e.g. `Project → hasResult` → publications, datasets, software). |

### Rendering notes that bite

- **Embeds render property rows and ignore `label`.** A composed `label` only
  shows where a type renders as `link`/`label`. To show a value-object's identity
  (e.g. `MonetaryAmount`), use `render: "label"`, not `embed`.
- **Long object lists.** Any relation with >100 objects starts collapsed to a
  count (`N values → Show first 100`) and never renders more than 100 rows — the
  tail is walked by opening a specific object. `groupByType` overrides this with
  per-type sections (each capped the same way).

## Worked example

```jsonc
"http://data.europa.eu/s66#Project": {
  "sidebar": "pin",
  "label":       ["http://data.europa.eu/s66#title"],
  "groupByType": ["http://data.europa.eu/s66#hasResult"],
  "order": ["…title", "…abstract", "…keyword", … ]
},
"http://data.europa.eu/s66#GrantPayment": {
  "render": "embed",
  "embedVia": "http://data.europa.eu/s66#isRecipientOf",
  "label": [
    "http://www.w3.org/2000/01/rdf-schema#label",   // the amount ("171776 EUR")
    "http://data.europa.eu/s66#hasRecipient"        // → OrganisationRole → Organisation
  ],
  "hide": ["http://www.w3.org/2000/01/rdf-schema#label"]
}
```

## Compatibility

The format is **additive**: unknown fields are ignored, and every field is
optional, so older config files keep loading. New per-type behaviour is a new
optional `TypeConfig` field; new profiling data is a new optional top-level key
(or a new optional field on an existing `TypeProfile` / `TypeProperty`). A
`--resume` profiler run upgrades an older cache in place — an old counts-only
`properties[]` entry simply gains facts fields on the next run. An `endpoints[]`
entry may likewise be an inline object or a string slug interchangeably.
